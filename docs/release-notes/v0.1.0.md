# v0.1.0 — Foundation: Sentry-shaped SDK family over OTLP/HTTP

**Released:** 2026-06-06
**Compatibility:** First public release. No prior versions to migrate from.

## Highlights

- ✨ Five packages shipped together — `monitor-core`, `monitor-node`, `monitor-browser`, `monitor-react`, `monitor-cli`
- ✨ Sentry-shaped public API (`init`, `captureException`, `captureMessage`, `withScope`, `setUser`, `setTags`, `addBreadcrumb`, `flush`, `shutdown`) on Node + browser
- ✨ First-class Express, Fastify, and NestJS bindings inside `monitor-node`
- ✨ React bindings: `<NeithlyErrorBoundary>`, `useNeithlyScope`, react-router v6 navigation breadcrumbs
- ✨ CLI: `monitor releases create` + `monitor sourcemaps upload` with SHA-256 dedup
- ✨ End-to-end QA pass against a live `neithly-monitor` backend — 4 matrices, ~5 s POST-to-SPA propagation
- 🔧 Auto-instrumentation by default: `uncaughtException`, `unhandledRejection`, fetch, XHR, console, HTTP
- 🐛 DSN parser fails fast at `init` — `DSN_MALFORMED` surfaces synchronously rather than silently dropping events for hours

## Known limitations (→ v0.1.1)

- 🐛 `service.name` MUST match the project slug or the backend silently drops records (200, zero persisted). Documented loudly + workaround in `init({ serviceName })`. See [Finding 01](../qa/findings/01-service-name-mismatch.md).
- 🐛 DSN bearer is the full plaintext, not the parsed `publicKey` — the field is misleadingly named. Internal SDK state holds `input`; rename to `ParsedDsn.bearer` planned for v0.1.1. See [Finding 02](../qa/findings/02-dsn-bearer-shape.md).
- 🐛 DSNs with non-empty `allowedOrigins` reject Node-side fetches (no `Origin` header). Workaround: mint server DSNs with empty `allowed_origins`. Backend UX split into Browser/Server DSN flows planned for v0.1.1. See [Finding 03](../qa/findings/03-allowed-origins-vs-node.md).
- ⏭️ `monitor-browser` v0.1 logs DSN-revoke failures but does not surface them — deferred follow-up.
